# API Guidelines

## Authority

The OpenAPI schema generated by FastAPI is authoritative for endpoints currently implemented. This document specifies target conventions and MUST NOT duplicate full request/response schemas by hand.

## Versioning

- Public application routes use `/api/v1` until a breaking contract requires a new version.
- Additive fields SHOULD be optional during migrations.
- Stable machine-readable error codes MUST accompany human-readable messages.

## Target ingest resources

```text
POST /api/v1/ingest-jobs
GET  /api/v1/ingest-jobs/{job_id}
POST /api/v1/ingest-jobs/{job_id}/retry
POST /api/v1/ingest-jobs/{job_id}/cancel
```

Job creation returns `202 Accepted`. Status is initially polled; Server-Sent Events MAY be added without removing polling.

## Errors

Target error shape:

```json
{
  "error": {
    "code": "TRANSCRIPT_PROVIDER_BLOCKED",
    "message": "Transcript provider rejected the request.",
    "stage": "fetching_transcript",
    "retryable": true,
    "request_id": "...",
    "details": {}
  }
}
```

Secrets, raw provider payloads, and stack traces MUST NOT appear in public error details.

## Streaming

Chat MAY use Server-Sent Events after non-streaming structured generation is verified. Streaming MUST end with a final validated answer envelope; partial tokens are not accepted citations.

## Client types

The TypeScript client SHOULD be generated from committed or reproducibly generated OpenAPI. Frontend types MUST NOT be maintained as a separate handwritten copy of backend schemas.
